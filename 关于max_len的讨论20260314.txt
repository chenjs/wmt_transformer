❯ config.py中，max_len参数是做什么用的                                          

⏺ Searched for 1 pattern, read 1 file (ctrl+o to expand)

⏺ max_len 参数用于限制序列的最大长度。

  从代码第19行可以看到：
  max_len: int = 54  # Based on 90th percentile of cleaned data (45 words * 1.2)

  具体作用：
  1. 限制输入序列长度 - 编码器收到的源语言句子不会超过54个 token
  2. 限制输出序列长度 - 解码器生成的目标语言句子不会超过54个 token

  这个值54是基于数据集的分析得出的：取清理后数据长度的90百分位（45词），乘以1.2
  的缓冲系数得到。这是一种常见的做法，在保证覆盖绝大多数样本的同时，避免处理极少
  数超长句子带来的内存和计算开销。

❯ “编码器收到的源语言句子不会超过54个 token”：                                  
这是在哪里的代码中限制的，你想详细地查找一下

⏺ max_len 参数在以下两处进行限制：

  1. src/data/batch.py:71-77 - 创建 batch 时截断

  src_tokens = [
      src_tokenizer(s, add_bos=False, add_eos=True)[:max_len]  # 第72行

  在 tokenizer 返回的 token 列表上直接用 [:max_len] 切片，截断超过54的部分。

  2. src/data/dataset.py:118-119 - collate_fn 中截断

  src_batch.append(item['src'][:max_len])  # 第118行

  同样在 collate_fn 中对已 tokenize 的序列进行截断。

  ---
  所以限制的方式是：在数据送入模型之前，直接对 token 列表做 Python 切片
  [:max_len]，丢弃超出长度的 token。这是比较简单直接的做法。

  ├────────┼────────────────────┼────────┼───────────────────────────────────┤
  │ 预处理 │ word数（空格分词） │ 100    │ 原始文本中用空格分隔的词数        │
  ├────────┼────────────────────┼────────┼───────────────────────────────────┤
  │ 训练时 │ token数（BPE编码） │ 54     │ subword tokenizer 编码后的 token  │
  │        │                    │        │ 数                                │
  └────────┴────────────────────┴────────┴───────────────────────────────────┘

  问题所在

  一个英文句子可能有 30-40 个词，但用 BPE tokenizer 编码后可能产生 40-80 个
  token（因为有些词会被拆分成多个 subword）。

  所以：
  - 预处理过滤：句子长度 ≤ 100 词
  - 训练截断：token 序列 ≤ 54 tokens

  这意味着：
  1. 预处理阶段无法保证 token 数 < 54
  2. 训练时的截断 [:max_len] 是必要的，会丢弃超出 54 个 token 的部分

  实际效果

  假设一个 50 词的句子：
  - 英文：50 词 → 约 60-75 个 token → 训练时被截断到 54
  - 德文：50 词 → 约 70-100 个 token → 训练时被截断到 54

  所以训练时的 max_len=54 截断是真正起作用的限制，预处理阶段的 max_words=100
  更像是一个粗略的初步过滤。

❯ 如果不限制token长度，会有什么问题？                                           

⏺ 不限制 token 长度会有以下问题：

  1. 内存爆炸（主要问题）

  Transformer 的 Self-Attention 计算复杂度是 O(n²)：

  序列长度 n    Attention 内存占用
  -----------  ------------------
  54           54²  =  2,916
  100          100² = 10,000  (3.4倍)
  200          200² = 40,000  (13.7倍)
  500          500² = 250,000 (85倍)

  对于 batch_size=12，内存占用还要乘以12。

  2. 计算速度大幅下降

  注意力矩阵变大后，每次前向传播和反向传播的计算量都大幅增加。

  3. 长距离依赖难以学习

  即使不限制长度，Transformer 也难以有效建模超长序列（几百个 token
  以上的依赖关系），这本身是 Transformer 架构的局限性。

  ---
  为什么当前设置是合理的

  max_len=54 是一个工程折中：
  - 覆盖绝大多数训练数据（基于90百分位）
  - 保持可接受的内存和计算开销
  - 实践中翻译任务大多数句子也确实不长

  如果要处理更长序列，通常的做法是：
  - 分段处理（将长句子分成多个片段）
  - 使用更高效的 attention 变体（如 Flash Attention、Linear Attention）
